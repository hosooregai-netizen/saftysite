# Feature Hardening Sprint

## 목적

source recovery가 build-safe MVP fallback에 그치지 않도록, 실제 사용자 경험 기준으로 기능을 고도화한다.

## Sprint 순서

1. mailbox hardening
2. photo-album hardening
3. headquarters-sites hardening
4. webhard regression gate
5. report/billing/auth gate
6. docs/registry update

## 완료 기준

- 기능이 단순히 build만 되는 상태가 아니라 명세의 핵심 UX를 충족한다.
- fallback component가 실제 기능 component로 대체되거나 충분히 보강된다.
- QA route smoke와 visual QA를 통과한다.
