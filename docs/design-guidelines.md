# ERP UI 디자인 기준서

## 1. 목적

이 문서는 이 프로젝트에서 새 UI를 만들거나 기존 화면을 수정할 때 사용하는 기본 디자인 기준서다.  
이 문서는 더 이상 `표 리스트`만을 위한 규칙이 아니라, 앞으로 반복해서 참고할 `운영형 ERP 화면 기준`을 정리하는 문서로 사용한다.

적용 대상은 아래와 같다.

- 현장 선택, 보고서 목록, 관리자 목록 화면
- 보고서 작성/편집 워크스페이스
- 일정, 사진첩, 메일함, 보조도구 같은 운영 화면
- 대시보드, KPI, 요약 카드, 상태 배너, 모달, 드로어
- 검색, 필터, 정렬, 일괄 작업, 업로드, 다운로드 같은 조작 영역

예외가 필요하면 화면 단위로 이유를 남긴다.  
기본값은 항상 이 문서를 따른다.

디자인 작업 중 이 문서에 없는 새 UI 요소, 새 화면 구조, 새 상호작용 기준이 필요해지면 바로 사용자에게 질문해 수정 틀을 먼저 잡는다.  
이때는 임의로 세부 규칙을 확정하지 않고, 무엇을 새 기준 항목으로 올릴지부터 합의한다.

## 2. 제품 성격

이 프로젝트의 UI는 홍보형 서비스가 아니라 `작업 중심 ERP`에 가깝다.

- 사용자는 빠르게 보고, 판단하고, 수정하고, 다음 작업으로 넘어가야 한다.
- 화면은 화려함보다 `정리된 정보`, `짧은 조작 경로`, `낮은 실수 가능성`이 중요하다.
- 같은 정보를 여러 번 반복하지 않고, 필요한 데이터만 앞에 둔다.
- 콘텐츠 영역은 차분하고 단정해야 하며, 브랜드 감도는 전역 헤더 같은 제한된 영역에서만 드러낸다.
- 관리자/관제 화면은 `데스크톱 우선`, 작업자 화면은 `데스크톱 우선 + 모바일 대응`으로 본다.

핵심 목표는 아래 네 가지다.

1. 첫 화면에서 지금 봐야 할 것과 해야 할 것이 바로 보일 것
2. 데이터를 비교하기 쉽고 입력하기 쉬울 것
3. 화면마다 구조가 달라지지 않고 예측 가능할 것
4. 부수 설명 없이도 자연스럽게 사용할 수 있을 것

## 3. 현재 코드에서 기준으로 삼을 공통 구조

현재 구현된 화면 구조를 기준 패턴으로 삼는다.

| 기준 패턴 | 실제 기준 | 의미 |
| --- | --- | --- |
| 전역 셸 | `WorkerAppHeader`, 사이드바, `WorkerShellBody` | 상단 공통 헤더 + 좌측 내비게이션 + 본문 작업 영역 |
| 섹션 기본 단위 | `sectionCard`, `sectionHeader`, `sectionBody` | 모든 목록/설정/운영 패널의 기본 골격 |
| 목록 제어부 | `TableToolbar`, `SortableHeaderCell` | 검색, 필터, 정렬, 내보내기, 건수 표시의 기준 |
| 편집 워크스페이스 | `WorkspaceShell` | 긴 문서 편집 화면의 헤더, 진행률, 섹션 이동, 본문 편집 구조 |
| 보조 상세 보기 | `AppModal` | 삭제 확인, 세부 미리보기, 2차 편집의 기준 |
| 요약 영역 | 요약 카드, KPI 카드, `SummaryBar`, `statCard` 계열 | 본문 전에 현재 상태를 압축해서 보여주는 레이어 |

새 화면을 만들 때는 이 구조를 확장하는 방식으로 접근하고, 화면마다 다른 패턴을 새로 발명하지 않는다.

## 4. 먼저 기준을 잡아야 할 요소

지금 문서가 표 중심이었다면, 앞으로는 아래 요소까지 같이 기준을 잡아야 한다.

| 요소 | 무엇을 정해야 하는가 | 기본 방향 |
| --- | --- | --- |
| 페이지 셸 | 전역 헤더, 사이드바, 본문 폭, 배경 톤 | 헤더는 강하게, 본문은 차분하게 |
| 페이지 헤더 | 제목, 뒤로가기, 현재 컨텍스트 | 제목은 짧게, 설명은 최소화 |
| 요약 레이어 | 상단 KPI, 카운트, 상태 요약 | 3~4개 이내 핵심 값만 노출 |
| 섹션 헤더 | 섹션 제목, 검색, 필터, 생성, 내보내기 | 전역 액션은 헤더 우측에 모음 |
| 목록 표현 | 표로 볼지 카드로 볼지 | 비교는 표, 미리보기 중심은 카드 |
| 검색/필터/정렬 | 어떤 조작을 즉시 노출할지 | 검색은 바로, 세부 필터는 패널로 |
| 행/셀 정보 구조 | 어떤 정보를 전면에 둘지 | 한 셀 한 정보, 메타는 최소화 |
| 상세/드릴다운 | 모달, 드로어, 별도 페이지 기준 | 짧은 작업은 모달, 긴 작업은 별도 페이지 |
| 폼/에디터 | 입력 그룹, 필수값, 저장 흐름 | 작업 순서대로 묶고 보조 설명은 줄임 |
| 상태/피드백 | 로딩, 빈 상태, 성공, 오류, 경고 | 짧고 명확하게, 위치는 일관되게 |
| 반응형 | 모바일에서 무엇을 숨기고 무엇을 유지할지 | 정보 계층 유지, 액션 우선순위 유지 |

이 표에 없는 새 요소가 실제 작업 중 나타나면 그 항목은 바로 사용자와 상의해 `새 기준 항목`으로 승격할지 결정한다.  
즉, 화면을 먼저 밀어붙이는 것이 아니라 `무엇을 문서 기준으로 추가할지`부터 질문하고 수정 범위를 정한다.

## 5. 공통 원칙

### 5-1. 정보는 한 번만 보여준다

- 같은 의미의 데이터를 제목 아래 보조문장으로 반복하지 않는다.
- 이미 상단 KPI에서 보여준 정보는 표에서 다시 긴 문장으로 풀지 않는다.
- 이미 페이지네이션이나 건수 레이블에 있는 숫자를 헤더 칩으로 다시 강조하지 않는다.
- 정적 부수 설명 문구는 기본적으로 넣지 않는다.
- `...만 보여줍니다`, `...기준으로 추려서 보여줍니다`, `...여기에서 조정합니다`, `...바로 반영됩니다` 같은 설명문은 금지한다.
- 정말 필요한 경우만 툴팁, 오류/경고 배너, 확인 모달처럼 사용자 행동과 직접 연결된 방식으로 분리한다.

### 5-2. 사용자는 읽기보다 스캔한다

- 첫 줄만 봐도 의미를 파악할 수 있어야 한다.
- 한 셀 안에 여러 줄 메타를 쌓는 패턴은 최소화한다.
- 조작 요소가 데이터 읽기를 방해하면 안 된다.
- 중요한 값은 위치와 정렬로 강조하고, 색과 장식으로 해결하려 하지 않는다.

### 5-3. 액션은 단계별로 분리한다

- 페이지 전체 액션은 페이지 헤더 또는 섹션 헤더에 둔다.
- 목록 전체 액션은 섹션 헤더 우측에 둔다.
- 관리자 리스트 탭은 검색창도 섹션 헤더 우측으로 올리는 것을 기본값으로 본다.
- 목록 헤더 아래 툴바에는 버튼을 두지 않는다.
- 목록 헤더 아래 툴바도 가능하면 만들지 않고, 꼭 필요할 때만 입력형 필터 전용으로 제한한다.
- `사업장`, `현장`, `사용자`, `콘텐츠`, `일정` 같은 관리자 CRUD 리스트는 같은 헤더 패턴으로 맞춘다.
- `엑셀 내보내기`, `업로드`, `자동생성`, 전역 토글 버튼은 섹션 헤더로 올린다.
- 필터가 2개 이상이거나 조합이 길어지면, 헤더에 `검색창 + 필터 버튼 + 주요 액션` 구조를 우선 적용한다.
- 사업장, 현장, 담당자, 기간, 계약유형처럼 범위를 좁히는 옵션은 헤더 필터 버튼 안으로 접는 것을 기본값으로 본다.
- 행 단위 액션만 행 버튼/행 메뉴에 둔다.
- 삭제, 발송, 연결 해제처럼 되돌리기 어려운 액션은 반드시 확인 단계를 둔다.

### 5-4. 본문은 차분하고 실무적으로 유지한다

- 브랜드 감도는 상단 헤더까지만 허용한다.
- 데이터 패널, 폼, 표, 모달은 장식보다 명확한 경계와 정돈된 밀도를 우선한다.
- 둥근 pill, 과한 그림자, 카드 남발, 과도한 색상 강조는 피한다.

## 6. 레이아웃 시스템

### 6-1. 페이지 기본 구조

모든 주요 화면은 아래 순서를 기본으로 삼는다.

1. 전역 헤더
2. 좌측 탐색 또는 보조 메뉴
3. 페이지 헤더
4. 상단 요약 레이어
5. 메인 작업 섹션

페이지 헤더에는 아래만 둔다.

- 화면 제목
- 이전 화면으로 가는 링크가 필요하면 1개
- 현재 문맥을 이해하는 데 필요한 최소 정보

페이지 헤더의 뒤로가기는 공통 텍스트형 컨트롤로 통일한다.

- 배경색이 깔린 버튼형을 기본값으로 쓰지 않는다.
- 스트로크가 있는 사각 버튼 구조를 기본값으로 쓰지 않는다.
- 기본 스타일은 배경 없음, 외곽선 없음, 텍스트 + 꺾쇠 한 개 조합이다.
- hover는 색 변화와 밑줄 정도만 허용하고, 강조 박스는 예외 상황에서만 쓴다.

페이지 헤더 아래 설명 문단은 기본적으로 넣지 않는다.
섹션 제목 아래에도 사용법 설명이나 화면 요약 문장을 붙이지 않는다.

### 6-2. 섹션 카드

목록, 설정, 운영 도구, 대시보드 패널은 모두 `섹션 카드` 구조를 기본으로 한다.

- `sectionHeader`: 제목, 현재 범위, 주요 액션
- `sectionBody`: 실제 데이터, 표, 폼, 카드 목록
- 필요하면 섹션 상단에 오류/알림 배너를 먼저 배치

섹션 하나는 하나의 작업 목표를 가져야 한다.  
한 카드 안에 `요약 + 목록 + 편집 + 안내문 + 로그`를 모두 넣지 않는다.

### 6-3. 요약 카드

- 페이지 상단 요약 카드는 3~4개 이내로 제한한다.
- 값, 라벨, 짧은 메타 정도만 둔다.
- 표와 같은 데이터를 다시 길게 설명하지 않는다.
- 운영 우선순위를 보여주는 값만 올린다.
  - 예: 배정 현장 수, 검색 결과 수, 미처리 건수, 최근 작업, 누락 건수

### 6-4. 분할 레이아웃

다음 화면은 분할 레이아웃이 적절하다.

- 메일함: 목록 + 상세
- 사진첩: 필터/그리드 + 상세 모달
- 편집 워크스페이스: 섹션 탐색 + 편집 본문
- 운영 도구: 요약/탐색 + 대상 목록

왼쪽은 선택과 탐색, 오른쪽은 상세와 작업에 집중한다.

### 6-5. 사이드탭 레이아웃

같은 도구 안에서 모드를 전환하는 탭이 3~6개 정도로 고정되어 있으면, 상단 버튼 묶음보다 좌측 사이드탭을 기본값으로 본다.

- 사이드탭은 섹션 제목 아래 본문 좌측 레일에 둔다.
- 섹션 헤더에는 제목, 검색, 주요 액션만 남기고 탭은 섞지 않는다.
- 라벨은 한 줄로 짧게 유지하고, 설명 문장이나 건수는 붙이지 않는다.
- 활성 상태는 카드 배경 변화와 작은 포인트 표시로 구분하고, 과한 색 채움 버튼처럼 보이지 않게 한다.
- 받은편지함/보낸편지함/연결 계정처럼 동일 도구 안의 고정 모드 전환은 이 패턴을 우선 적용한다.
- 화면 폭이 좁아지면 사이드탭은 본문 위로 올라갈 수는 있지만, 여전히 같은 레일 묶음으로 유지한다.

## 7. 화면 유형별 기준

### 7-1. 진입/목록 화면

대표 화면:

- `/`
- `/sites/[siteKey]`
- `/admin`의 사용자/현장/보고서/일정 목록

기본 구조:

1. 페이지 제목
2. 핵심 요약 카드
3. 섹션 헤더
4. 검색/필터/정렬
5. 목록

기준:

- 첫 진입에서 다음 행동으로 연결되는 버튼이 보여야 한다.
- 요약 카드는 현재 범위를 설명하고, 목록은 실제 작업 대상을 보여준다.
- 목록의 부수 메타는 줄이고, 클릭 후 상세에서 보게 한다.

### 7-2. 대시보드/관제 화면

대표 화면:

- `/admin`
- `/admin?section=overview`
- `/admin?section=analytics`

기준:

- 카드와 표는 역할을 분리한다.
- 카드에서는 이상 징후, 누락, 마감, 추세를 빠르게 보여준다.
- 표에서는 실제 처리 대상과 우선순위를 보여준다.
- Top 리스트, 경고 패널, 알림 패널은 “정보량”보다 “조치 우선순위”를 우선한다.

### 7-3. 편집 워크스페이스

대표 화면:

- `/sessions/[sessionId]`
- `/sites/[siteKey]/quarterly/[quarterKey]`
- `/sites/[siteKey]/bad-workplace/[reportMonth]`

기준:

- 긴 문서 편집 화면은 별도 워크스페이스로 분리한다.
- 상단에는 현재 문서 제목, 진행 상태, 문서 액션만 둔다.
- 본문은 섹션 단위로 나누고, 한 번에 한 작업 덩어리만 읽히게 한다.
- 저장/동기화/문서 생성 오류는 화면 상단 또는 툴바 근처의 고정된 위치에서 알려준다.
- 사진 첨부, 메타 수정, 문서 다운로드는 편집 본문과 분리된 제어영역에서 처리한다.

### 7-3-1. 불량사업장 신고 워크스페이스

대표 화면:

- `/sites/[siteKey]/bad-workplace/[reportMonth]`

기준:

- 섹션 순서는 `원본 기술지도 보고서 선택 -> 취약 사항 선택 -> 통보서 기본 정보 -> 첨부 표 -> 내부 메모`를 기본으로 유지한다.
- `원본 보고서 선택`과 `취약 사항 선택` 단계에서 문서 7 지적사항과 문서 4 후속조치 결과를 같이 보여, 문서 편집 전에 가져올 항목을 먼저 걸러낼 수 있어야 한다.
- 통보서 기본 정보 상단에는 `자동`, `검토`, `수동` 구분을 먼저 보여 준다. 사용자가 어떤 값까지 믿고 내려가면 되는지 즉시 판단할 수 있어야 한다.
- 현장/본사 마스터, 지도일, 첨부서류처럼 근거가 분명한 값만 자동 반영한다.
- 공정률, 상단 이행 확인일처럼 파생 계산이나 행 단위 데이터를 한 값으로 접는 항목은 `검토 후 유지` 항목으로 다룬다.
- 담당 요원 연락처, 대표자, 지방노동청(지청)장처럼 근거 데이터가 없거나 관할 매핑이 없는 항목은 빈값 + 플레이스홀더를 기본으로 두고, 현재 로그인 사용자 정보나 임의 추정값으로 채우지 않는다.
- 상단 `이행 확인일`은 선택한 행들의 확인일이 한 값으로 일치할 때만 자동 반영한다. 행별 확인일이 섞이면 상단에서는 직접 검토 상태로 남겨 둔다.

### 7-4. 운영 보조 화면

대표 화면:

- `/calendar`
- `/sites/[siteKey]/photos`
- `/mailbox`

기준:

- 작업 순서가 짧고 분명해야 한다.
- 내부 모드 전환이 있으면 상단 가로 탭보다 좌측 사이드탭을 먼저 검토한다.
- 필터와 현재 대상 범위가 먼저 보여야 한다.
- 빈 상태에서도 “왜 비어 있는지”가 바로 이해되어야 한다.
- 리스트가 아닌 카드/그리드를 쓰더라도, 액션 위치와 정보 우선순위는 ERP 톤을 유지한다.

## 8. 데이터 표시 기준

### 8-1. 표와 카드의 선택 기준

표를 쓰는 경우:

- 여러 행을 비교해야 할 때
- 열 기준으로 스캔해야 할 때
- 상태, 날짜, 담당자, 수치가 반복될 때

카드를 쓰는 경우:

- 이미지 미리보기가 핵심일 때
- 메일 스레드처럼 제목/본문 일부/최근 시간 조합이 중요한 경우
- 모바일에서도 시각적 맥락이 더 중요할 때

`비교`가 핵심이면 표, `대상 미리보기`가 핵심이면 카드로 간다.

### 8-2. 표 디자인 원칙

- 기본은 `한 셀 = 한 정보`다.
- 셀 안 제목 아래 보조문장을 붙이는 2단 구조는 기본 금지다.
- 상태, 담당자, 날짜, 카테고리, 사업장/현장은 별도 열로 뺄 수 있으면 우선 분리한다.
- 행 전체가 상세 화면으로 이어지는 경우는 체크박스/행 메뉴 셀을 제외하고 행 전체를 클릭 가능하게 한다.
- 행 메뉴는 마지막 열에 고정하고, 읽기 흐름을 방해하지 않게 한다.
- 표 헤더는 짧고 명확해야 하며, 모호한 업무 용어를 줄인다.

셀 높이는 아래 기본값을 쓴다.

- 표 헤더 셀: 36~40px 수준
- 비교형 표의 일반 본문 셀: 44px 수준
- 표 안에서 바로 편집하는 입력/선택 셀: 44px 수준
- 문서 편집 리스트처럼 여유가 조금 더 필요한 본문 셀: 48px 수준
- 여러 줄 입력 셀: 시작 높이 88~96px

표 안에서 편집하는 입력 구조는 아래처럼 통일한다.

- 표 셀 안에 입력창 박스를 한 번 더 띄우지 않는다.
- 입력, 선택, 여러 줄 입력은 셀 자체에 바로 쓰는 느낌으로 보여야 한다.
- 셀 내부 컨트롤은 `border: 0`, `background: transparent`, `border-radius: 0`, `box-shadow: none`을 기본으로 한다.
- 포커스 강조는 입력창 자체보다 셀 배경 변화나 셀 inset focus로 처리한다.
- 예외는 검색창, 전역 필터, 모달 독립 폼처럼 표 바깥 단독 입력일 때만 허용한다.

문서형 리스트/표의 글자 크기는 아래 기준으로 통일한다.

- 문서 내부 블록 제목(표 위 제목): `22~24px`, `700`, line-height `1.3`
- 표 헤더 셀, 라벨 셀, 차트 패널 제목행, 요약 라벨행: `14px`, `700`, line-height `1.35`
- 값 셀 본문, 입력값, 일반 설명 본문: `15px`, `400~500`, line-height `1.45~1.6`
- 보조 상태, 보조 액션, 보조 설명: `12~13px`, `500~600`

같은 문서 안에서는 같은 역할의 제목 크기를 섞지 않는다.

- 한 문서 안의 표 헤더 셀만 `14px`, 일부 차트 제목행만 `16px`처럼 키우지 않는다.
- `표/리스트 제목 셀`과 `본문 위 블록 제목`은 한 단계 차이만 둔다.
- 볼드, 자간, 색 차이로 강조를 더하더라도 기본 크기 단계는 유지한다.

같은 행 안에서는 셀 높이를 섞지 않는다.

- 같은 행에 라벨 셀, 값 셀, 선택 셀이 함께 있으면 가장 자주 쓰는 높이 하나로 맞춘다.
- 한쪽은 44px, 다른 쪽은 58px처럼 섞어서 행 리듬이 흔들리게 만들지 않는다.
- 예외는 이미지, 긴 텍스트, 업로드처럼 본질적으로 높이가 커야 하는 경우만 허용한다.

### 8-3. 검색, 필터, 정렬

- 검색은 가장 자주 쓰는 조작이므로 즉시 노출한다.
- 필터는 기본 1차만 보이고, 세부 필터는 패널이나 메뉴로 묶는다.
- 버튼처럼 보이는 전역 필터는 헤더에 두고, 본문 툴바에는 셀렉트형 필터를 우선한다.
- 관리자 목록 화면은 `제목 + 검색창 + 필터 버튼 + 기능 버튼`을 한 줄 헤더 기준으로 정리한다.
- 사업장/현장/담당자/기간처럼 여러 조합이 필요한 필터는 보고서 화면처럼 헤더 필터 메뉴로 묶는다.
- 인라인 셀렉트가 길게 늘어져 섹션 헤더와 분리되면 구조를 다시 접는다.
- 필터 메뉴, 정렬 메뉴, 행 메뉴는 카드나 표 경계에 잘리지 않도록 표 바깥 레이어로 떠야 한다.
- 정렬은 `숫자/날짜/이름/제목`처럼 선형 비교가 자연스러운 컬럼에만 둔다.
- 상태, 유형, 담당자, 심각도처럼 분류형 컬럼은 정렬보다 필터 우선이다.
- 이미 필터에서 좁히는 컬럼은 정렬 마크를 숨기는 쪽이 낫다.

### 8-4. 건수와 상태 표시

- 건수는 과장된 칩보다 일반 텍스트에 가깝게 보여준다.
- 상태는 색만으로 구분하지 않고 반드시 텍스트를 같이 둔다.
- 색 강조는 `오류`, `경고`, `주의`, `성공`, `완료`처럼 의미가 분명한 경우만 쓴다.
- 일반 메타는 무채색 계열로 유지한다.

### 8-5. 날짜, 숫자, 단위

- 날짜는 화면 밀도에 따라 `YYYY-MM-DD` 또는 `MM.DD`로 줄일 수 있다.
- 시간은 24시간 형식을 기본으로 한다.
- 숫자는 단위와 함께 보여준다.
  - 예: `12건`, `3명`, `24%`
- 금액과 수치는 비교가 쉬워야 하므로 열 정렬과 자릿수 일관성을 우선한다.

### 8-6. 차트와 그래프 스트로크

- 차트 스트로크는 정보 우선순위에 따라 두께를 나눈다.
- 보조 그리드선, 기준선, 구분선: `1px`
- 축선 또는 기준축 강조선: `1~1.5px`
- 주요 데이터 라인: `2.5~3px`
- 보조 비교 라인: `2px`
- 포인트 마커 외곽선: `2px`

같은 차트 안에서는 스트로크 단계를 과하게 늘리지 않는다.

- 한 차트 안에서 실질적으로 쓰는 굵기는 `1px`, `2px`, `3px` 정도의 3단계 안에 정리한다.
- 강조하려는 데이터보다 보조선이 더 두꺼워지면 안 된다.
- 장식 목적의 두꺼운 외곽선, 과도한 그림자, 불필요한 이중 스트로크는 쓰지 않는다.

차트 유형별 기본값은 아래를 따른다.

- 라인 차트: 데이터선 `3px`, 보조 비교선 `2px`, 그리드선 `1px`
- 막대 차트: 막대 외곽선은 기본적으로 생략하고, 꼭 필요할 때만 `1px`
- 도넛/파이 차트: 조각 경계선은 기본 생략, 꼭 필요할 때만 배경색 기준 `1px`
- 포인트/원형 마커: 채움색 + 외곽선 `2px` 조합을 기본으로 한다.

## 9. 폼과 편집 기준

### 9.1 폼 구조

- 입력은 업무 순서대로 묶는다.
- 관련 있는 필드는 같은 행 또는 같은 카드로 배치한다.
- 데스크톱에서는 2열 그리드를 기본으로 하고, 모바일에서는 1열로 내린다.
- 긴 설명 텍스트는 필드 아래에 기본적으로 두지 않는다.
- 사용자가 실수하기 쉬운 필드에만 힌트를 넣는다.

### 9-2. 입력 컨트롤

- 기본 컨트롤 높이는 36~38px 수준으로 맞춘다.
- 버튼, 입력창, 셀렉트는 같은 밀도와 같은 모서리 톤을 유지한다.
- 텍스트 입력, 날짜 입력, 선택 입력의 시각 톤은 통일한다.
- 체크박스/라디오/토글은 의미가 분명한 위치에만 둔다.
- 표 셀 안에 직접 들어가는 입력/선택 컨트롤은 독립 박스처럼 띄우지 말고, 셀 높이 기준에 맞춰 셀 자체에 붙여 보이게 한다.
- 여러 줄 입력 영역의 보조 액션은 별도 버튼 바보다 입력셀 내부 하단 우측의 텍스트 액션을 우선한다.

### 9-3. 파일/사진 업로드

- 업로드 박스는 높이를 고정해 레이아웃이 흔들리지 않게 한다.
- 미리보기가 핵심이면 콘텐츠가 넘치지 않도록 `contain` 또는 고정 비율로 정리한다.
- 업로드/삭제/미리보기는 같은 박스 안에서 해결하되, 주 액션은 하나만 강조한다.
- 문서 편집 화면에서 사진 자체가 본문인 경우에는 표형 리스트로 바꾸지 말고 카드형을 유지한다.
- 다만 ERP 문맥에서 고정 슬롯이 반복되는 화면은 카드보다 리스트 표형 블록을 우선한다. 카드형은 추가/삭제가 잦은 가변 영역에서만 예외적으로 쓴다.
- 사진 카드의 내부 순서는 `라벨/삭제 액션 -> 사진 -> 한 줄 메타 입력`을 기본으로 한다.
- 고정 슬롯형 사진 표에서는 삭제를 제목 셀에 반복 노출하기보다 사진 셀 우상단의 작은 텍스트 오버레이로 둘 수 있다.
- 고정 슬롯형 사진 표에서는 사진을 넣어도 기존 셀 높이와 표 전체 높이가 늘어나지 않아야 하며, 이미지는 셀 내부에서 `object-fit: contain` 기준으로 축소되어 잘리지 않게 맞춘다.
- 기술지도 문서 전반의 사진 슬롯은 문서 본문 높이와 서로 연동해 늘어나지 않게 하고, 슬롯별 고정 높이를 유지한다.
- 사진 카드는 데스크톱 기준 내부 패딩 `12px`, 요소 간 간격 `8~10px`을 기본으로 하고, 사진 박스 높이는 `220~260px`, 모바일은 `190~220px` 범위로 맞춘다.
- 사진 카드의 하단 메타 입력은 별도 두꺼운 입력 박스보다 얇은 1줄 입력 형태를 우선한다. 기본 높이는 `36~40px`로 맞춘다.
- 삭제는 빨간색 텍스트 액션으로 작게 두고, 사진보다 시각적으로 앞서지 않게 한다.
- ERP 통일감을 우선할 때는 카드 그림자를 최소화하고 `얇은 보더 + 옅은 헤더 배경 + 흰 본문` 구조를 기본으로 한다.
- 비교가 목적이 아닌 순차 사진 기록은 데스크톱에서도 `1열 스택`을 허용한다. 이때 카드 폭은 넓히고 사진 높이는 낮춰서 스크롤 밀도를 맞춘다.
- 사진 카드의 라벨 행은 표 헤더처럼 연한 회색 배경을 쓰고, 메타 입력 영역은 본문보다 한 단계만 옅은 배경으로 정리한다.
- 주요 입력 칸은 기본적으로 흰 배경을 유지하고, 제목·라벨 행만 연한 배경색으로 구분한다.
- 차트가 포함된 ERP 섹션도 카드보다 `헤더 행 + 흰 본문`의 표형 패널을 우선한다.

### 9-4. 긴 편집 화면

- 진행률, 섹션 이동, 문서 액션은 상단 툴바에 모은다.
- 본문 카드 안에는 현재 섹션 제목과 그 섹션 조작만 둔다.
- 저장, 생성, 동기화는 본문보다 상위 레벨에서 제어한다.
- 본문 상단 `notice`는 성공/실패/차단 사유처럼 지금 행동에 직접 영향을 주는 상태만 다룬다.
- 설명을 위한 notice는 만들지 않는다.

### 9-4-1. 문서형 자동 채움 기준

- 문서형 편집 화면의 입력값은 `자동 반영`, `검토 후 유지`, `수동 입력` 세 종류로 나눈다.
- `자동 반영`은 현장 마스터, 본사 마스터, 선택한 원본 보고서처럼 source-of-truth가 분명할 때만 허용한다.
- `검토 후 유지`는 계산값, 요약값, 여러 행에서 접어 올린 상단 메타처럼 사용자가 마지막으로 한 번 더 판단해야 하는 값에 적용한다.
- `수동 입력`은 대표자, 관할 기관명, 담당자 연락처처럼 현재 데이터 모델에 근거가 없는 항목에 적용한다.
- 수동 입력 항목은 임의 기본값보다 빈값과 짧은 플레이스홀더를 우선한다.
- 자동 채움 구분 안내는 필드마다 긴 설명을 반복하지 않고, 섹션 상단의 짧은 리뷰 카드나 상태 배너로 한 번에 정리한다.

## 10. 상세 보기, 모달, 드로어 기준

### 10-1. 모달

모달은 아래 상황에서 쓴다.

- 삭제 확인
- 짧은 생성/수정
- 이미지/파일/레코드 상세 미리보기
- 현재 화면을 벗어나지 않아도 되는 2차 작업

모달 안에는 새 탐색 구조를 만들지 않는다.

- 모달 구현 기본은 `AppModal`을 쓴다.
- 날짜 선택, 상태 선택, 짧은 범위 지정처럼 `2~3개 필드만 잠깐 고르는 작업`도 별도 `dialog`를 새로 만들지 말고 같은 모달 구조로 맞춘다.
- 선택 모달은 화면 중앙 정렬을 기본으로 하고, 제목 1개 + 선택 필드 1~3개 + 하단 `취소/확인` 액션만 둔다.
- 선택 모달 안에 추가 설명 문장, 보조 네비게이션, 별도 툴바를 넣지 않는다.
- 선택 필드는 일반 입력/셀렉트와 같은 톤을 유지하고, 모달 자체만 특별한 비주얼로 꾸미지 않는다.

### 10-2. 드로어

드로어는 아래 상황에서 쓴다.

- 모바일에서 사이드 메뉴 대체
- 보조 탐색
- 공간을 많이 쓰는 필터 패널

드로어는 본문 작업보다 우선하면 안 된다.

### 10-3. 별도 페이지

아래는 별도 페이지로 분리한다.

- 긴 문서 작성
- 다단계 편집
- 저장/검토/출력 흐름이 있는 작업
- 참조 관계가 복잡한 작업

## 11. 상태와 피드백 기준

### 11-1. 로딩

- 목록은 스켈레톤 또는 최종 구조와 유사한 자리 표시 형태를 쓴다.
- 전체 화면 스피너보다 섹션 단위 로딩이 낫다.

### 11-2. 빈 상태

- 한 문장으로 끝낼 수 있으면 한 문장으로 끝낸다.
- “왜 비어 있는지”가 중요하면 이유 1개만 덧붙인다.
- 행동이 필요한 경우 버튼 1개만 같이 둔다.

### 11-3. 오류와 알림

- 오류 배너는 작업이 끊기는 위치 바로 위에 둔다.
- 알림 배너는 짧고 조치 가능한 문장으로 작성한다.
- 성공 메시지는 지나치게 강한 색보다 중성 배너로 처리한다.

### 11-4. 확인 메시지

- 삭제/해제/발송 전 확인 모달에는 대상 이름을 명시한다.
- 결과를 되돌릴 수 없는 경우만 추가 설명을 붙인다.

## 12. 시각 톤과 토큰 기준

현재 구현 토큰을 기준으로 아래 톤을 유지한다.

- 기본 팔레트: 네이비, 슬레이트, 화이트, 연한 회청색
- 강조색: 골드 계열은 브랜드나 제한된 강조에만 사용
- 위험색: 붉은 톤은 삭제, 오류, 경고에만 사용
- 본문 배경: 밝고 차분한 중성 톤
- 콘텐츠 패널: 흰색 기반, 약한 테두리, 약한 그림자

세부 기준:

- radius: 컨트롤 1~2px, 카드/패널 4px 안팎
- 과하게 둥근 pill 형태는 기본 금지
- 본문 패널의 그림자는 얕고 짧아야 한다
- 본문 글자 기본 크기는 14px, UI 텍스트는 12~13px, 패널 제목은 18px 전후
- 숫자 KPI는 22~30px 범위에서 제한적으로 사용한다

전역 헤더만 예외적으로 더 강한 시각 톤을 가져갈 수 있다.

## 13. 반응형과 접근성 기준

### 13-1. 반응형

- 관리자용 데이터 화면은 데스크톱에서 가장 완성도 높게 설계한다.
- 모바일로 내려갈 때는 열을 줄이고, 설명을 줄이고, 액션을 위로 올린다.
- 모바일에서도 제목, 현재 범위, 주 액션, 선택 상태는 반드시 유지한다.
- 요약 카드는 2열까지 허용하되, 의미 없는 카드 증식은 금지한다.

### 13-2. 접근성

- 클릭 가능한 행은 키보드 포커스도 가능해야 한다.
- 색만으로 상태를 전달하지 않는다.
- 포커스 링은 숨기지 않는다.
- 행 클릭과 체크박스/행 메뉴 클릭 영역은 충돌하지 않게 분리한다.
- 아이콘만 있는 버튼은 `aria-label` 또는 동등한 텍스트 맥락이 있어야 한다.

## 14. 현재 프로젝트 화면군별 우선 적용 포인트

| 화면군 | 대표 라우트 | 우선 적용 기준 |
| --- | --- | --- |
| 현장 진입/목록 | `/`, `/sites/[siteKey]` | 상단 요약 + 간결한 목록 + 빠른 다음 액션 |
| 보고서 목록 | `/sites/[siteKey]`, `/admin?section=reports` | 검색 우선, 상태 정렬 제거, 행 전체 클릭, 메타 축소 |
| 보고서 편집 | `/sessions/[sessionId]`, 분기/불량작업 에디터 | 워크스페이스 툴바, 섹션 탐색, 고정된 상태 피드백 |
| 운영 도구 | `/calendar`, `/photos`, `/mailbox` | 분할 레이아웃, 현재 범위 표시, 빠른 보조 액션 |
| 관리자 관제 | `/admin`, `overview`, `analytics` | KPI와 처리 목록 분리, 경고 우선순위 강조 |

### 14-1. `전체 보고서`에 우선 반영할 기준

- 정렬은 `보고서`, `기준일`, `수정일` 중심으로만 유지한다.
- `유형`, `현장`, `담당자`, `상태`, `발송`, `품질체크`는 정렬보다 필터 우선으로 전환한다.
- 검색, 필터, 내보내기, 업로드는 별도 상단 툴바가 아니라 섹션 헤더 기준으로 정리한다.
- 건수는 한 곳에서만 보여준다.
- 보고서 행은 체크박스와 메뉴 셀을 제외하고 전체 클릭 가능하게 만든다.
- 불필요한 2차 메타 줄은 제거한다.

### 14-2. 관리자 리스트 탭에 우선 반영할 기준

- 사용자, 사업장, 현장, 콘텐츠, 일정 리스트의 전역 버튼은 모두 섹션 헤더로 올린다.
- 리스트 본문 상단에는 검색, 셀렉트, 날짜 입력만 남긴다.
- 버튼형 필터가 본문에 남아 있으면 셀렉트형 필터 또는 헤더 액션으로 다시 정리한다.

## 15. 작업 체크리스트

새 화면을 만들거나 수정할 때 아래를 확인한다.

1. 이 화면의 1차 목적이 한 문장으로 설명되는가
2. 기존 문서에 없는 새 요소나 새 기준 항목이 생겼다면, 사용자에게 먼저 질문해 수정 틀을 잡았는가
3. 전역 구조와 섹션 구조가 현재 공통 패턴과 맞는가
4. 첫 화면에서 지금 봐야 할 정보와 해야 할 행동이 바로 보이는가
5. 상단 요약, 목록, 상세가 같은 정보를 반복하고 있지 않은가
6. 이 화면은 표가 맞는가, 카드가 맞는가
7. 검색/필터/정렬이 역할별로 분리되어 있는가
8. 셀 안에 불필요한 2줄 메타가 들어가 있지 않은가
9. 상태/오류/빈 상태 문장이 짧고 명확한가
10. 모달과 별도 페이지의 경계가 적절한가
11. 모바일에서 정보 계층과 주 액션이 유지되는가
12. 클릭 영역, 포커스, 라벨이 접근성 기준을 만족하는가

이 문서의 기본 방향은 한 문장으로 요약하면 아래와 같다.

> 강한 브랜드 헤더 위에, 차분한 데이터 본문을 얹고, 필요한 정보와 행동만 남기는 ERP UI를 만든다.
